# 從多個不完美訊號取一個值：反脆弱偵測原則

> 一份可分享的設計原則，通用於任何「要從幾個都不完全可靠的來源推導出一個值」的情境，不限本專案。
> 姊妹篇：[如何確認一個改動「真的更好」](verification-principles.md)。

## 核心一句話

> **當一個值要靠多個不完美的訊號推導時，把它們分層，讓任一訊號缺失或改變時只是「少一票」，而不是整個崩盤；並設一個永遠不會被跌破的硬下界。**

robust（穩健）只是「壞了不會更糟」；**antifragile（反脆弱）是「從變動中獲益」** —— 訊號越多樣，覆蓋越完整；少了一個來源，系統自動降級到下一個，使用者甚至不會察覺。

---

## 為什麼不要單壓「最權威」的那個訊號

直覺反應是：找到一個更權威、更即時的訊號，就改成只信它。這是**脆弱**的：

- 那個訊號可能改名、改格式、某版本後消失（你不掌控的上游）。
- 它可能只在某些情況出現（例如只在某種請求、某種帳號）。
- 一旦它失效，你又退回原點 —— 甚至更糟，因為舊的後備路徑已被你刪掉。

反脆弱的作法不是「換一個更好的單一來源」，而是**把新來源加進投票，同時保留舊來源當後備**。新來源好的時候它贏；它消失的時候，系統只是少一票。

---

## 三個機制

### 1. 多訊號分層（有就用，沒有不退步）

把所有能推出該值的訊號列出來，每個都只是「一票」，誰都不是唯一真相：

```
value = pick(
   權威但可能變動的訊號   ,   // 有就優先用
   會落後但穩定的舊訊號   ,   // 當加分票，不當唯一真相
   永遠存在但較粗的訊號   ,   // 兜底
)
```

### 2. 硬下界（via negativa 的單調保證）

有些事實是**不可違反的物理約束**，任何訊號都不能讓結果低於它。把它寫成 `max(..., 下界)`，並讓推得的「升級」狀態**單調、不回退**。這一條保證：無論上面的投票怎麼亂，輸出永遠在合理範圍內。

### 3. 能力閘門（防過度宣稱）

「某能力是否啟用」的旗標，常常是**帳號/客戶端級**的，會出現在不該套用的個體上。所以旗標要配一個**能力閘門**：只有「本來就支援該能力的個體」才套用。否則一個全域旗標會把所有個體都誤標成有該能力。

---

## via negativa：移除，不是疊加

反脆弱常常意味著**拿掉機制**而非加機制。

實例：在拿到「不落後的每筆訊號」之前，我們本來規劃一個 session 級的「黏著狀態」來記住升級結果、避免閃爍。但一旦證實新訊號**每一筆都在**，那個黏著狀態就變成多餘的狀態管理 —— 於是刪掉它。**少一個會出錯的狀態，就少一個故障點。** 先有證據，再決定要不要那個機制，而不是先把機制都加上。

## 判斷力：什麼時候值得這樣做

| 用 | 不用 |
|---|---|
| 值來自你不掌控的上游訊號（header、檔案格式、第三方欄位） | 值來自你自己、穩定、單一的來源 |
| 訊號會隨版本漂移、可能消失 | 契約穩定、有 SLA |
| 有一個無法違反的物理下界可當保險 | 沒有任何可信的兜底，硬湊反而製造假確定性 |
| 多個來源彼此獨立、能互補覆蓋 | 來源彼此完全相關，多列只是錯覺 |

過度套用也有成本：每多一個訊號就多一條要測的路徑。只有當**單一來源真的會讓你被上游變動咬到**時，多層才划算。

## 實例：context window 大小的偵測（#58 / PR #60）

ccxray 要顯示每回合的 ctx%，分母是該 model 的 context window。原本只信 system-prompt 裡的 model marker，但它在 session 中途切換 model 後會**落後好幾個 turn**，導致分母用錯、ctx% 從 ~99% 一回合崩到 ~20%。

修法不是「換成另一個更權威的單一訊號」，而是分層：

| 單一訊號失效 | 系統反應 |
|---|---|
| `anthropic-beta: context-1m-*` header 改名／被上游拿掉 | 降回 `[1m]` system marker ＋ usage 下界 |
| `[1m]` marker 永遠不來（title-gen 等請求） | header ＋ usage 接手 |
| header 與 marker 都沒有（舊版 client／其他工具） | model 名稱表 ＋ usage 下界＝原本行為 |
| 觀測 usage 還很小 | header 立刻給出正確窗口，不必等 usage 衝破門檻 |

- **硬下界**：顯示的窗口永遠不小於已觀測用量（`max(window, usageInferred)`），所以絕不會出現「用量 > 上限」的假 99%。
- **能力閘門**：`context-1m` header 是帳號級旗標，連 haiku title-gen 請求也帶；用 `SUPPORTS_1M` 限定 opus/sonnet 才套用，否則 haiku 會被誤標成 1M 窗口。
- **via negativa**：因為 header 每筆都在，原規劃的 session-sticky 黏著狀態被移除。

新增一個 1M model 家族＝在 `SUPPORTS_1M` 加一行，不是改判定邏輯 —— 這就是「從變動中獲益而非受損」。

---

## 反模式

- **單壓新訊號**：發現一個更權威的來源就刪掉所有後備，只信它。上游一變你就裸奔。
- **沒有硬下界**：純靠投票，沒有不可違反的約束兜底，異常輸入時輸出可以是任意值。
- **旗標不配閘門**：把帳號級能力旗標直接當「此個體有此能力」，造成過度宣稱。
- **先加狀態再說**：還沒證明需要，就先加 session 黏著／快取／回填等狀態機，每一個都是未來的故障點。
