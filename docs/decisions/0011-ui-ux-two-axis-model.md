# ADR 0011 — UI/UX 分軌：二維模型

> 2026-07 定案。決策過程：模擬五人專家團（Karpathy / Willison / Wattenberger / Beck / Rauch，各自查近半年觀點、獨立評估、再辯論）收斂。

## Context

UI/UX 相關 issue 在自動化 pipeline 中常被當成單一類別處理。實務上有些 UI 改動可以完全用機械斷言驗收（token 計數、三態切換、CLS 數值），有些則只有人眼能判（手感、間距、層級和諧）。混為一談會導致兩種失敗：該自動的卡在人工佇列白等，該人看的被綠燈放過。

## Decision

**UI/UX 不是一個類別**，用兩條正交軸分類：

### 軸一：定義成熟度（WHAT 清不清楚）

issue 的規格是否足以讓任何人（或 agent）獨立判斷「做對了沒」。

- 定義不清 → **一律不自動**。Agent 會自編定義、寫出「對自選定義成立」的測試——Beck 的 genie 作弊源頭：測試綠燈但實際沒解決問題。

### 軸二：驗收可判定性（DONE 能否機器判）

驗收動作屬命題層還是知覺層。

- **命題層**（對比 / CLS / token / 三態 / 計數）→ 貪婪自動化。結果有客觀對錯，assertion 能判定。
- **知覺層**（手感 / 間距 / 層級 / 和諧 / 佈局）→ 判決永遠是 owner 的眼睛。Gate green 只證明「函式輸出符合斷言」，不證明「畫面上使用者看到的正確」。

### 四象限

|  | 驗收可機械判（命題） | 驗收需人眼（知覺） |
|---|---|---|
| **定義清楚** | 全自動：agent 做 + gate 驗 | agent 做、驗收要人看 |
| **定義不清** | 不自動（genie 風險） | 不自動（雙重不確定） |

### 執行原則

1. **減法優先**：先把 UI issue 拆到命題層能吃的部分（數值、三態、對比），盡量縮小知覺層範圍。
2. **證據閘**：動 `public/` 的 PR 必附 before/after 截圖或操作錄影——機器產證據，人判對錯。
3. **AI 批評當提示不當裁決**：AI 可以指出「這個間距看起來偏大」，但最終判決是 owner 的眼睛。
4. **IA/新視覺模式永不自動**：資訊架構和全新呈現模式需要人的設計判斷，不屬自動化範圍。

## 頭號失敗模式（五人一致）

**綠燈冒充「看起來對」。** 自動化的價值是「把看起來對拆成無法作弊的斷言 + 把實況端到人面前」，不是代替人判斷。

## Consequences

- Issue triage 需判兩軸再決定走哪條路徑（見 `docs/triage.md` 維度 A/B）。
- 知覺層 issue 走 `pipeline:needs-visual-review` 終端 gate，PR merge 前 owner 必須看畫面。
- 命題層 issue 可全自動驗收，不必等人看。
- 混合型 issue（如 lane 歸屬：邏輯可單測 + 呈現需看畫面）以知覺層為準——先判驗收面。
