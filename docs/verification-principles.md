# 如何確認一個改動「真的更好」

> 一份可分享的驗證原則。通用於任何程式改動,不限本專案。

## 核心一句話

> **「測試全綠」只證明沒變差,不證明更好。**
> 「沒人反對」也只證明沒人看到問題。兩者都不是「改善」的證據。

要主張「更好」,你得拿出一個**能分辨新舊的檢查**。

---

## 最佳做法:把那個檢查「先寫出來、先看它紅」

```
✅ 最佳:  寫紅 → 改 → 看綠
⚠️ 次佳:  改 → 補測試 → 事後倒回舊碼證明它會紅
```

差別在**順序**,而順序決定三件事:

1. **逼你先定義「更好」。** 你寫不出那個會失敗的檢查,除非已想清楚「成功長什麼樣」。先寫紅,強迫你動手前就把「better」講具體,而不是事後拿「測試綠了」來合理化。
2. **證據自動到手。** 你親眼看著它從紅變綠——這就是「舊碼會失敗」的硬證據,不必事後補。
3. **堵死最常見的錯誤。** 「改完才驗、用全綠當改善」根本分不出「修好了」和「本來就沒壞」。先寫紅就不可能犯這個錯。

這就是 TDD 的 red-green,但重點不在「測試先行」這個儀式,而在它的**驗證副產品**:你免費得到「舊碼會失敗」的證明。

### 配一條判斷力

> **純重構不寫紅。** 如果你要做的是「不改行為」的重構,卻發現自己能寫出一個會在舊碼失敗的測試——停下來,你其實改了行為。那是警訊,不是成就。

---

## 可信度階梯(由弱到強)

```
最強  ┌─ 差異檢查:同一檢查在舊碼 FAIL、新碼 PASS
      │     → 證明「修了一個真實存在的差異」,不是換寫法的 churn
      ├─ 客觀指標:可量化的數字往好的方向動
      │     → 重複數 6→1、p95 延遲 ↓、副作用數=0、覆蓋率 ↑
      ├─ 獨立第二意見:沒看過你推理的人/工具也沒挑出問題
      │     → 證「不只你覺得對」,但仍只是「沒看到問題」
最弱  └─ 自己跑全綠
            → 只證「沒破壞既有行為」(no regression)
```

多數人停在最下面兩層,然後宣稱「更好」。盡量往上爬一層。

---

## 改動類型決定驗證方式(最關鍵的一張表)

| 改動性質 | 正確的確認方式 | 反模式 |
|---|---|---|
| **修 bug / 改行為** | 差異檢查(舊碼**必須**失敗) | 只跑新碼全綠就說修好 |
| **純重構 / 不改行為** | 綠燈不變 + **客觀結構指標** | 寫一個會 fail-on-old 的測試——有它就代表行為被改了、不純 |
| **效能優化** | 同條件 before/after 基準,多次取中位數 | 跑一次、看體感 |
| **文件 / 提示詞** | 正確性可驗;有效性靠 with/without 對照或事後「是否復發」 | 「寫了就會更好」 |

> 核心洞察:**純重構「不該」有差異測試。** 如果有,代表你以為的重構其實改了行為——那是警訊,不是成就。

---

## 操作原則

1. **先重現,再修。** 修之前先讓 bug 以「失敗的測試 / 可見的錯誤」重現。這同時保證你手上有那個「old fail」——否則你根本不知道修的是不是對的東西。
2. **斷言「結果」,不斷言「實作」。** 測使用者層級的可觀察結果,不是內部細節。對著實作寫測試,重構就會無謂亮紅燈,逼你不敢清理技術債。
3. **測「不該發生的事」。** 別只測 happy path;測負面案例與邊界。很多 bug 藏在「沉默地沒發生」裡。
4. **沉默不是成功。** 監看 / 檢查若只比對「成功訊號」,崩潰時會跟「還在跑」長得一樣。覆蓋**所有終態**(成功+各種失敗)。
5. **客觀指標 > 主觀宣稱。** 能量化就量化:重複數、副作用數、行數、覆蓋率、延遲分位數。
6. **第二意見要真獨立,且批判採納。** 找沒看過你推理的人/工具複核;對方只是附和你的框架就是回音不是驗證。別盲從——把它當同事,不當權威。
7. **誠實標示驗證的邊界。** 明說你驗了什麼、**沒**驗什麼。隱瞞缺口比缺口本身更危險。
8. **規則寫完不是終點。** 出現更好的洞察就回去校準既有規則/文件,否則它會與你的新認知漂移。

## Fallback:已經 shipped 才補證明(差異測試)

如果你忘了先寫紅、code 已經改完,用拋棄式 worktree 把證明補回來:

```bash
# 1. 在改動前的 commit 開一個拋棄式 worktree
git worktree add /tmp/before <改動前的commit>

# 2. 把「新」測試複製進舊碼
cp test/my-new.test.js /tmp/before/test/

# 3. 在舊碼上跑 —— 它應該要 FAIL
cd /tmp/before && <跑測試>
#    舊碼 FAIL + 新碼 PASS = 你真的修了東西 ✅
#    舊碼也 PASS           = 這測試分辨不出新舊,證明不了「更好」⚠️

# 4. 清理
git worktree remove --force /tmp/before
```

---

## 一句話總結

> 改完別只問「測試過了嗎?」,要問:
> **「有沒有一個檢查,在舊版會失敗、在新版會通過?」**
> 最好是動手前就寫好它、看它紅。有,你就握有「更好」的硬證據;沒有,你頂多證明了「沒弄壞」。
